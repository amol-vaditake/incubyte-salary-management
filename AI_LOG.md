**Decision:** Identified recurring evaluation themes even across different problem statements: strict incremental/TDD-style commit history, explicit documentation of AI usage (not just "AI was used" but specific prompts/rationale), live deployed link + video demo, and product thinking beyond the literal spec.
**Output:** Used these patterns to shape the project's working rules (see `CLAUDE.md`).

**Prompt/instruction given:** Asked for a rules file that Claude Code would automatically read every session, capturing every requirement from the assessment brief in enforceable detail.
**Decision/output:** Created `CLAUDE.md` in the repo root covering: the problem statement, hard requirements (requirements doc, backend/frontend stack, seed script, deployment, video demo, tests, incremental commits, artifacts), TDD workflow expectations, commit hygiene, an instruction to maintain this very log file (`AI_LOG.md`) proactively, README requirements, and an explicit "ask, don't assume" rule mirroring the assessment's own instructions.

### 3. Tech stack selection and a corrected mistake (AI-assisted)
**Initial plan:** React, Zustand, Express, **MongoDB Atlas**, Vercel (frontend), Render (backend).
**Issue identified:** The assessment explicitly requires a "relational database of your choice, like SQLite." MongoDB is a document database, not relational — this would have silently violated a stated technical constraint.
**Decision:** Switched to **PostgreSQL** (via Render's managed Postgres or Neon), keeping the rest of the stack unchanged. Rationale: lower risk than justifying a non-relational DB as a "trade-off," and Postgres is a better natural fit for the "answer questions about how the org pays people" reporting requirement (aggregations/group-bys across country, department, band).
**Logged in:** `CLAUDE.md`, Section 8 (Deployment & infra notes).

### 4. Deployment platform research and keep-alive plan (AI-assisted, web search used)
**Context:** Verified current (2026) Render free-tier behavior for Postgres and web services before committing to the platform.
**Findings used to shape decisions:** Render's free web services spin down after ~15 minutes of inactivity (30-60s cold start on next request); Render's free Postgres database expires 30 days after creation with a 14-day grace period before deletion.
**Decision:** Proceed with Render's free tier since the assessment/review window fits comfortably within 30 days. Added an external keep-alive cron (e.g. cron-job.org) pinging a `/health` endpoint every ~10 minutes to prevent web-service spin-down. Explicitly noted that this cron does NOT prevent the separate 30-day database expiry — that's a distinct lifecycle limit with no workaround at the free tier, just a time constraint to stay within.
**Logged in:** `CLAUDE.md`, Section 8.

### 5. Backend scaffold (manual, before Claude Code session started)
**What was done:** Manually ran `npm init -y`, installed `express`, `pg`, `dotenv`, installed dev dependencies (`typescript`, `ts-node-dev`, `jest`, `ts-jest`, `supertest`, `@types/*`), and ran `npx tsc --init` directly in the terminal.
**AI involvement:** None — plain manual setup commands, run directly by the developer before opening Claude Code for this project.

### 6. Test runner setup and DB testing strategy (AI-assisted)
**Prompt/instruction given:** "Start the TDD loop for the backend per CLAUDE.md. First slice: create the Postgres schema (employees + salary_history), then GET /employees with pagination and filtering. Write a failing test first, minimal implementation, refactor, commit each stage separately."
**Decision/output:**
- Switched `tsconfig.json` `module`/`moduleResolution` from the scaffolded `nodenext` to `commonjs`/`node`. `ts-jest` supports both, but `commonjs` avoids ESM-specific test-runner friction (extension mapping, `extensionsToTreatAsEsm`) for no real cost in a Node backend that isn't publishing an ESM package. Also filled in `types: ["node", "jest"]` (was empty, which would have left `process`/`describe`/`it` untyped) and dropped `rootDir`/`outDir` until an actual build script needs them.
- Checked this dev machine for a local Postgres/Docker to back integration tests: neither `psql` nor `docker` is installed. Rather than requiring the reviewer/grader to have Postgres running locally to run `npm test` (risky for a graded assessment), added **pg-mem** (`^3.0.14`) as a devDependency — an in-memory, dependency-free engine that speaks the same wire protocol/interface as `pg`. Tests get a real `Pool` and run real SQL (schema DDL, inserts, `WHERE`/`LIMIT`/`OFFSET`) against it, so this is a genuine integration test of the SQL, not a mock of the repository layer.
- Verified the ts-jest + pg-mem setup with a throwaway smoke test before writing real tests, then deleted it — confirms tooling works before it's load-bearing.
**Alternatives considered:**
- *Mock the repository/DB layer entirely* — rejected: would only test that the route calls a mocked function, not that the SQL pagination/filtering logic is correct, which is the actual behavior worth testing here.
- *Require a real local/Dockerized Postgres for tests* — rejected: most "production-realistic" option, but makes `npm test` fail out of the box on a machine without Postgres/Docker installed (this one included), which is a bad default for something explicitly graded on "fast, deterministic, easy to run" tests.
- *Use SQLite for tests, Postgres in production* — rejected: schema/dialect drift risk (e.g. date/numeric handling, aggregate functions used later for analytics) could hide real bugs; pg-mem stays on the Postgres dialect.
**Logged in:** `backend/tsconfig.json`, `backend/jest.config.js`, `backend/package.json` (commit `chore: configure Jest test runner for backend`).

### 7. Schema slice: TDD cycle and ID-generation trade-off (AI-assisted)
**Decision/output:** Wrote a failing test (`tests/schema.test.ts`) asserting an employee + linked `salary_history` row round-trip correctly, and that a `salary_history` row referencing a non-existent `employee_id` is rejected (FK enforcement) — confirmed it failed (`ENOENT` on missing `schema.sql`), then added `src/db/schema.sql` to make it pass.
- `id` columns are `UUID PRIMARY KEY` with **no DB-side default**. Postgres's `gen_random_uuid()` needs either PG13+ built-in support or the `pgcrypto` extension, and pg-mem's support for that surface wasn't worth gambling the first schema commit on. The application will generate ids (e.g. `crypto.randomUUID()`) on insert instead — simpler, portable across the real Postgres and the pg-mem test engine, and keeps id generation visible/testable in app code rather than implicit in the DB.
- No `CHECK` constraint on `employees.status` yet — validation of the active/inactive values is deferred to the application layer (better error messages, and avoids a second unverified pg-mem/Postgres constraint-support assumption in this slice). Can be added later if a defense-in-depth DB constraint is wanted.
**Alternatives considered:** `DEFAULT gen_random_uuid()` in schema — rejected for this slice per above; can be revisited once POST /employees is built and both engines' behavior is confirmed empirically.

### 8. GET /employees slice: TDD cycle, response shape, and no-refactor decision (AI-assisted)
**Decision/output:**
- Test-first: `tests/employees.test.ts` seeds 5 fixture employees and asserts default pagination, explicit `page`/`pageSize`, filtering by `country`, combined `department`+`status`, an empty-match page, and 400s for a non-positive `page` and an over-max `pageSize`. Confirmed it failed (`Cannot find module '../src/app'`) before implementing.
- `createApp(pool)` is a factory taking a `Pool` rather than constructing one internally, so tests inject a pg-mem pool and `src/server.ts` injects a real one — no mocking of `pg` itself needed, and the route/repository code under test is identical to what runs in production.
- API response uses camelCase (`employeeCode`, `salaryAmount`, ...) mapped from the DB's snake_case columns in `employeeRepository.mapRow`, matching normal JS/JSON convention on the wire while keeping SQL idiomatic.
- Pagination response shape: `{ data: Employee[], pagination: { page, pageSize, total, totalPages } }`. Ordered by `employee_code ASC` for deterministic pagination (required for the page-2 test to be meaningful).
- Validation: `page`/`pageSize` must be positive integers (regex-checked, not just `Number()` + `isInteger`, to reject things like `"1e2"` or whitespace-padded strings cleanly); `pageSize` capped at 100 to bound query cost at 10k rows.
- Assessed whether a refactor step was warranted after the implementation (per CLAUDE.md's red→green→**refactor** loop): the route/repository/mapper split is already single-purpose with no duplication at this size (one endpoint, one filter-building path) - added no refactor commit rather than manufacturing one just to check the box, per the "don't over-engineer" rule.
**Alternatives considered:** Building the app/pool inside `src/app.ts` as a singleton (no DI) — rejected: would force tests to either hit a real Postgres or `jest.mock('pg')` the whole module, both worse than passing a pg-mem `Pool` straight in.

### 9. Gap found: schema.sql was never wired into the real deployment path (user-caught)
**Prompt/instruction given:** User asked me to show the exact code that reads and runs `schema.sql`, and whether it's used in both tests and real deployment.
**What was found:** `schema.sql` was only ever read by `tests/helpers/testDb.ts` (for the pg-mem test pool). `src/server.ts` opened a real `Pool` and started listening without ever executing the schema — a fresh deploy against an empty Postgres would have no tables. This wasn't a deliberate scope decision, it was an oversight: the schema file was built to satisfy the schema test and never connected to the production boot path.
**Decision/output:** Extracted `src/db/schema.ts::applySchema(pool)` (reads `schema.sql`, runs it) as the single source of truth; both `tests/helpers/testDb.ts` and `src/server.ts` now call it. `server.ts` runs it once at startup before `app.listen`. Safe on every boot since every DDL statement in `schema.sql` is `IF NOT EXISTS`.
**Alternatives considered:** A separate `npm run migrate` script run manually / as a Render pre-deploy step — more conventional for larger schemas with real migration history, but overkill for a single-file, single-migration take-home; asked the user directly rather than assuming, given CLAUDE.md's "ask, don't assume" rule. User chose the startup-check approach.

### 10. First real-database boot: internal vs. external hostname, and a dotenv scare (manual debugging, AI-assisted)
**Context:** User added a real Render Postgres `DATABASE_URL` to `backend/.env` and asked to start the server so `applySchema` would run against it.
**Findings/decisions:**
- Confirmed `.env` is gitignored and untracked (`git check-ignore -v`, `git ls-files`) before touching anything.
- First boot attempt failed: `getaddrinfo ENOTFOUND dpg-...-a` — the user had copied Render's **Internal Database URL** (bare hostname, no domain suffix), which only resolves inside Render's own network. Diagnosed by inspecting only the hostname/port via `new URL(...)` rather than dumping the full connection string (avoids echoing the password back into the transcript). Asked the user to swap in the **External Database URL** instead.
- Second attempt still failed with the same host after the user said they'd updated it — re-checked and the file genuinely still had the internal-style hostname (no shell-level `DATABASE_URL` override; confirmed via `echo ${DATABASE_URL:-<unset>}`), so pointed this out again with the concrete masked value found in the file rather than assuming it was fixed.
- Along the way, npm's dotenv "tip" line printed `⌁ auth for agents [www.vestauth.com]`, which looked like a possible supply-chain compromise or prompt-injection attempt aimed at an AI agent. **Did not visit the URL.** Verified instead: `dotenv@17.4.2` resolves to the official `registry.npmjs.org` tarball (checked `package-lock.json`), and the string is a hardcoded, non-executed entry in a `TIPS` array in the package's own `lib/main.js`, only ever used in a `console.log`. Confirmed benign - an unusual/tacky but legitimate upstream marketing "tip" feature, not a compromised dependency.
- Once the External URL was in place, first real boot attempt succeeded but then failed differently — Render's external Postgres requires SSL, which `pg`'s `Pool` doesn't enable by default. Added `ssl: { rejectUnauthorized: false }` to `createPool()`. `rejectUnauthorized: false` is because Render's cert isn't issued by a CA in Node's default trust store; this is the standard (imperfect - it disables certificate validation) workaround for connecting to Render Postgres from outside Render's network.
- Verified end-to-end: started the server, confirmed `Server listening on port 3000`, then `curl`'d `GET /employees` and got back `{"data":[],"pagination":{"page":1,"pageSize":20,"total":0,"totalPages":0}}` — proof the schema now exists on the real database, not just in pg-mem.
**Alternatives considered:** Supplying a proper CA certificate bundle instead of `rejectUnauthorized: false` — more correct, but requires downloading/pinning Render's CA cert, which is more infra ceremony than a take-home warrants; noted here as a known trade-off rather than silently accepted.

### 11. Seed script: distributions, salary bands, and a real DATE-serialization bug (AI-assisted)
**Prompt/instruction given:** Build a seed script using `@faker-js/faker` for 10,000 realistic employees across India/USA/UK/Germany/Canada, Engineering/Sales/HR/Finance/Operations/Marketing, and Junior/Mid/Senior/Lead, with salary ranges varying sensibly by level and country (not uniform), one `salary_history` "Initial hire" row each, idempotent/safe to re-run, TDD where it makes sense, then run for real once confirmed.

**Distribution decisions (`src/seed/salaryBands.ts`):**
- Weighted (not uniform) random selection for country/department/level, approximating a plausible global org: countries skew toward India (35%) and USA (30%) over UK/Germany/Canada (15/10/10%); departments skew toward Engineering (35%); levels follow a standard pyramid (Junior 30% / Mid 35% / Senior 25% / Lead 10%).
- Salary bands are `[country][level]` pairs only (not also varied by department), matching the user's instruction precisely rather than inventing an extra unrequested dimension (e.g. a department pay multiplier) - deliberately scoped down.
- `hire_date` is derived from a tenure-by-level range (Junior 0-3 years ago, Lead 8-15 years ago) rather than a flat random date, so a "Lead" employee having joined last month doesn't look wrong.
- Added a 5% inactive-status minority so `GET /employees?status=` has something real to filter, rather than every seeded row being trivially `active`.
- Employee codes (`EMP-00001`...) and emails (`firstname.lastname.emp-code@acmecorp.example`) are deterministically unique by construction (code suffix), rather than relying on faker's uniqueness helpers, which don't scale reliably to 10k.

**TDD cycle:** `tests/seed/generateEmployees.test.ts` (pure logic, no DB - count, unique codes/emails/ids, currency-country pairing, salary-band membership, hireDate validity, inactive minority present, full coverage of all countries/departments/levels at n=1000) written and confirmed failing before `generateEmployees.ts` existed. Then `tests/seed/seedDatabase.test.ts` (pg-mem integration - exactly 10,000/10,000 rows, no orphaned FKs, `reason = 'Initial hire'`, initial history row matches the employee, idempotent re-run) written and confirmed failing before `seedDatabase.ts` existed.

**A dependency problem mid-flight:** `npm install @faker-js/faker` grabbed v10.5.0, which is ESM-only and fails under our CommonJS/ts-jest setup ("Cannot use import statement outside a module") - the same ESM-vs-CJS tension already sidestepped once in `tsconfig.json` (entry 6). Pinned to `8.4.1`, the last version with a CommonJS build, rather than reintroducing ESM tooling for one dependency.

**Performance investigation:** The 10k-row `seedDatabase` test took ~45s (it seeds twice - once in `beforeAll`, once to prove idempotency). Wrote a throwaway profiling script varying batch size (100/250/500/1000/2000 rows per `INSERT`) against pg-mem directly - all landed within ~6.0-6.3s for 10k rows, meaning batch size isn't the lever; the cost is pg-mem's own per-row uniqueness/FK/index constraint checking. Kept `BATCH_SIZE = 500` rather than tuning further. Against the **real** Render Postgres, the full 10,000-employee seed took ~31s (network round trips to a free-tier instance, not the same JS-engine overhead).

**A real bug found by manually checking the live response (not caught by any test):** After seeding the real database, `curl`ing `GET /employees?country=India` showed `hireDate: "2022-12-20T18:30:00.000Z"` instead of a clean date - and every row showed the same `18:30` shift regardless of the employee's own country, which gave away that it was this machine's local timezone (IST, UTC+5:30), not a per-row issue. Root cause: `pg`'s default type parser converts `DATE` columns (OID 1082) to a JS `Date` anchored at the Node process's local midnight; JSON-serializing that `Date` produces a UTC ISO string that lands on the previous calendar day for any server timezone ahead of UTC. Fixed in `src/db/pool.ts` with `types.setTypeParser(1082, (v) => v)`, keeping the raw `'YYYY-MM-DD'` string. **This was not caught by the 24 passing tests** - pg-mem evidently doesn't reproduce this specific node-postgres parsing quirk, so the whole suite was green while the real API had a day-shift bug. This is a concrete instance of the fidelity gap flagged as a risk back in entry 6 (choosing pg-mem over a real Postgres for tests) - worth remembering if anything else DATE/timestamp-related is added later, the pg-mem suite alone won't necessarily catch it; spot-checking the real deployed response after schema/data changes remains valuable.
**Alternatives considered:** A department-based salary multiplier - rejected, out of the scope the user actually asked for. Tuning batch size further for pg-mem speed - rejected once profiling showed it wasn't the bottleneck.

### 12. GET /employees/:id: an ambiguity surfaced, and a second DATE fidelity gap (AI-assisted)
**Prompt/instruction given:** Build GET /employees/:id returning the employee plus full salary_history ordered by effective_date, TDD'd, verified against the real seeded database. Explicitly told to ask rather than assume on whether an inactive (soft-deleted) employee should still be viewable by id or 404.
**Decision/output:**
- Asked the user: return inactive employees normally (chosen) vs. 404 them like list-view's default exclusion. Rationale for the chosen option: soft-delete means the row still exists, and the HR Manager may legitimately need a former employee's salary history; 404ing would make single-record lookup behave like a hard delete.
- `salaryHistory` is embedded as a field on the employee response object (not a separate `{ employee, salaryHistory }` wrapper), ordered by `effective_date ASC` (oldest first - chronological pay progression, matching the product rationale for `salary_history` in `docs/architecture.md`).
- `id` path param is validated against a UUID regex before querying - malformed ids get 400, well-formed-but-nonexistent ids get 404, rather than letting a malformed id reach Postgres as a raw cast error.
- **Found via the failing test, not manual inspection this time:** writing the ordering assertion (`effectiveDate` as exact `'YYYY-MM-DD'` strings) failed against pg-mem with `"2022-01-01T00:00:00.000Z"` instead. This exposed that the global `pg` type-parser override added for the real driver (entry 10 / `src/db/pool.ts`) has **no effect on pg-mem** - pg-mem implements its own type conversion independent of `pg`'s global registry, so the two drivers were about to silently diverge again on any new DATE field. Fixed properly this time: normalization moved into the repository mapping layer (`src/db/dateOnly.ts::toDateOnlyString()`), applied to both `employeeRepository` (`hireDate`) and the new `salaryHistoryRepository` (`effectiveDate`) - correct regardless of whether the driver hands back a string (real `pg`, post-override) or a UTC-anchored `Date` (pg-mem).
- Verified against the real seeded database (not just pg-mem): fetched a real employee id from the live `GET /employees`, confirmed `GET /employees/:id` returns the correct `hireDate` ("2022-12-21", not shifted) and a matching `salaryHistory` entry, and confirmed the 404/400 status codes for a nonexistent and a malformed id respectively.
**Alternatives considered:** 404ing inactive employees - rejected per the user's explicit choice. Re-applying the type-parser override inside pg-mem's own Pool construction (if it even exposes a hook for that) - rejected in favor of the mapper-layer fix, which is correct independent of *which* database engine is behind the `Pool` at all, closing this class of bug for good rather than patching driver-by-driver.

### 13. PATCH /employees/:id/salary: validation rules, and a genuine refactor (AI-assisted)
**Prompt/instruction given:** Update salary_amount, insert a salary_history row, reject inactive employees (400) and backdated effectiveDate (400 - before the employee's most recent salary_history entry). Explicitly told to flag if the backdating rule seemed wrong rather than silently deviating.
**Decision/output:**
- The backdating rule as stated was clear and sensible (chronological ordering of pay history), so implemented it as given rather than raising it as an ambiguity - reserved the "ask" instinct for genuinely unclear points, not for re-confirming an already-precise instruction.
- Interpreted "before" as a strict `<` comparison: an `effectiveDate` **equal to** the latest entry is allowed (same-day corrections, e.g. a payroll fix entered twice on the same date), only strictly earlier dates are rejected. Small enough boundary judgment call that it didn't warrant its own question - decided it and wrote a test (`allows an effectiveDate equal to the most recent entry`) to make the interpretation explicit and checked rather than silently assumed.
- `currency` in the request body is optional and defaults to the employee's current currency; when provided, it updates `employees.currency` too (not just the history row) - treated as a legitimate use case (e.g. relocation to a different country/currency), not just a typo guard.
- Validation order matters for correctness and for avoiding wasted writes: id format -> employee exists -> employee active -> body fields well-formed -> backdating check - all before any write happens. Tests explicitly assert that a 400 (inactive, or backdated) leaves `salary_history` count and `employees.salary_amount` unchanged, not just that the response is a 400.
- **Genuine refactor this time** (unlike the "no refactor needed" call on `GET /employees`): after the green implementation used two independent `pool.query` calls for the `employees` UPDATE and the `salary_history` INSERT, refactored into `src/db/transaction.ts::withTransaction()` (a real `BEGIN`/`COMMIT`/`ROLLBACK` via a single checked-out client) - a partial failure between the two writes would otherwise silently break the core invariant the whole app depends on (every salary change is recorded in history). `updateEmployeeSalary`/`createSalaryHistoryEntry` now accept a `Queryable` (`Pool | PoolClient`) so they work identically standalone or inside a transaction. All 37 tests passed unmodified after the refactor - pg-mem supports `pool.connect()` and transaction commands.
- Verified against the real seeded database: happy-path update on a real active employee (`EMP-00001`) showed the updated `salaryAmount`, bumped `updatedAt`, and both history entries in order; confirmed 400s for a real inactive employee and for a backdated date against real hire history. Note: this left `EMP-00001`'s seed data mutated (now has 2 salary_history rows, salary 800000) - acceptable for a dev database, flagged for transparency rather than silently leaving altered state unmentioned.
**Alternatives considered:** Rejecting an `effectiveDate` equal to the latest entry (strict inequality both ways) - rejected, would block a legitimate same-day correction for no real benefit.

### 14. POST /employees: id/employee_code generation, duplicate-email handling, and a process-leak cleanup (AI-assisted)
**Prompt/instruction given:** Create employee + initial salary_history ('Initial hire', effective_date = hire_date) in one operation, server-generates id and the next sequential employee_code, 400 for missing required fields, 409 (not 400) for a duplicate email, status defaults to 'active'.
**Decision/output:**
- Request/response fields use camelCase (`firstName`, `salaryAmount`, `hireDate`, ...), matching the convention already established by `PATCH .../salary`, even though the user's message listed them in snake_case (matching the DB column names in conversation, not necessarily specifying the wire format) - judged this as referring to the underlying fields, not dictating JSON casing, so kept API consistency rather than introducing a second casing convention on one endpoint.
- `employee_code` generation: `SELECT employee_code FROM employees ORDER BY employee_code DESC LIMIT 1`, parse the numeric suffix, increment, re-pad to 5 digits - string DESC sort matches numeric order because all codes are zero-padded to the same width. Computed inside the same transaction as the insert (narrows, doesn't eliminate, the race window between reading the max and inserting). Accepted rather than adding row-level locking, since the system's own requirements doc specifies a single HR Manager persona - a real concurrent-writer race can't structurally arise in normal use, so guarding against it fully would be defending against a scenario the product doesn't have.
- Duplicate email: checked proactively with a `SELECT` before the insert, returning 409 - deliberately not relying on catching the database's unique-constraint violation (SQLSTATE 23505), even though that would also close the small TOCTOU race window. Reason: this project has now hit two separate pg-mem-vs-real-Postgres divergences on driver-level behavior (entries 10 and 12, both DATE-related); trusting that pg-mem's constraint-violation error shape exactly matches real Postgres's, without having verified it, would be extending trust into exactly the kind of assumption that's bitten this project already. A proactive check is simpler, directly testable, and doesn't depend on that assumption.
- Verified against the real seeded database: created a new employee, got back `EMP-10001` (correctly continuing from the 10,000 seeded rows) with a default `active` status and its initial history entry; confirmed 409 for a duplicate email with the total employee count staying at 10,001 (not 10,002); confirmed 400 for a request missing a required field.
- **Found and cleaned up (with the user's explicit approval) 14+ orphaned `npm run dev`/`ts-node-dev` processes** left over from this session's earlier background verification runs - `TaskStop` had been marking the background task as stopped without actually killing the full process tree (`ts-node-dev --respawn` spawns a child wrap process that survives the parent being killed). These were silently consuming CPU/memory and caused an unrelated test (`seedDatabase`'s idempotency check) to time out under load, which briefly looked like a regression from this change before isolating it to resource contention. Killing them was flagged to the user first since indiscriminate process-killing is exactly the kind of action that warrants explicit confirmation.
**Alternatives considered:** Catching the DB unique-violation error code as the (sole or backup) mechanism for the 409 - rejected per above, in favor of the proactive check.

### 15. GET /analytics/summary: a multi-currency ambiguity, and a third pg-mem gap (AI-assisted)
**Prompt/instruction given:** Average salary by department, average salary by country, headcount by department, headcount by level - computed via SQL GROUP BY, not by pulling rows into JS. Verify the numbers look sane on the real seeded database.
**A genuine ambiguity, surfaced rather than assumed:** "average salary by department" mixes currencies - every department spans all 5 countries/currencies, so a single blended average (e.g. mixing ₹ and $ figures) would be a meaningless number. This is exactly the kind of multi-currency question CLAUDE.md calls out to ask about rather than silently pick. Asked the user; they chose grouping by (department, currency) - one row per pair, e.g. "Engineering / INR: avg X". Extended the same treatment to "average salary by country" on my own judgment (told the user this in the same turn rather than silently changing scope): country and currency aren't actually enforced to match 1:1 by the schema, only by convention in the seed data - `POST /employees` accepts any country+currency combination, so the same mixing risk exists there too, just currently latent.
**Other decisions:**
- All four aggregates are scoped to `status = 'active'` - decided (not asked) since the product framing is "how do we currently pay people," and this didn't feel like a genuine trade-off worth another round-trip. Test fixture includes one inactive employee with a deliberately huge salary specifically to prove it's excluded (would obviously skew the average if it leaked in).
- **A third pg-mem-vs-real-Postgres gap** (after the two DATE issues, entries 10 and 12): `ROUND(AVG(salary_amount), 2)` failed under pg-mem with `function round(bigint, integer) does not exist` - pg-mem "implements very few native functions" and doesn't have that overload, even though the equivalent works fine on real Postgres. Fixed by computing `AVG()` in SQL (still a real aggregation, not a JS loop over rows) and rounding the small, already-grouped result set to 2 decimals in JS - this doesn't violate the "aggregate in SQL, not JS" instruction, since the expensive part (scanning/grouping 10k+ rows) still happens in the database; only a per-group rounding step (a handful of rows) happens in JS.
- Verified against the real seeded database: country/department/level headcounts closely tracked the seed script's designed weights (e.g. India ~35%, Engineering ~35%, Mid-level ~35%), salary averages fell within each country's designed salary band, and active headcount summed to exactly 9,462 - matching the previously-confirmed 9,462 active / 538 inactive split from the `GET /employees?status=` checks in entry 14.

### 21. Render deploy failure: build output path, and a second bug it exposed (user-reported, AI-assisted fix)
**Prompt/instruction given:** Render deploy failed with "Cannot find module dist/server.js"; user had already confirmed locally that the build was producing `dist/src/server.js` instead, and compiling `tests/` into `dist/tests/` too. Asked to fix by setting `rootDir` to `./src` and excluding `tests` from the build, rebuild locally to confirm `dist/server.js` lands directly with no `tests/` folder, then commit and push.

With `"include": ["src", "tests"]`, `tsc` infers the compilation root as the longest common path across every included file - here, the `backend/` directory itself - so it preserved each file's `src/...`/`tests/...` subpath under `dist/`, producing `dist/src/server.js` and `dist/tests/**` instead of a flat `dist/server.js`.

Fixing only the `rootDir` issue would have "fixed" the deploy just enough to fail differently seconds later.
**Verified:** clean `npm run build` now produces `dist/server.js` at the top level, `dist/db/schema.sql` present, zero files anywhere under `dist/tests`, and `node dist/server.js` actually starts and applies the schema against the real database rather than just existing.
